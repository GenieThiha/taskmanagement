# Data Model

All tables use **PostgreSQL 16**. All PKs are UUID generated by `gen_random_uuid()`. All tables include `created_at` and `updated_at` (`TIMESTAMPTZ NOT NULL`) managed automatically by Sequelize hooks. Soft-delete where noted — never hard-delete records.

---

## Entity Relationship Overview

```
users ──< projects (owner_id)
users ──< tasks    (assignee_id, reporter_id)
projects ──< tasks (project_id)
tasks ──< comments (task_id)
users ──< comments (author_id)
users ──< notifications (recipient_id)
tasks ──< notifications (reference_id where reference_type = 'task')
```

---

## users

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PK, DEFAULT gen_random_uuid() | Unique user identifier |
| email | VARCHAR(255) | UNIQUE, NOT NULL | Login email address |
| password_hash | VARCHAR(255) | NOT NULL | bcrypt hash (cost factor 12) — never expose in API responses |
| full_name | VARCHAR(100) | NOT NULL | Display name |
| role | ENUM | DEFAULT 'member', NOT NULL | `admin` \| `manager` \| `member` |
| is_active | BOOLEAN | DEFAULT TRUE, NOT NULL | Soft-delete flag |
| failed_login_attempts | SMALLINT | DEFAULT 0, NOT NULL | Reset to 0 on successful login |
| locked_until | TIMESTAMPTZ | NULLABLE | Set to NOW() + 30 min after 5 failed attempts |
| created_at | TIMESTAMPTZ | NOT NULL | — |
| updated_at | TIMESTAMPTZ | NOT NULL | — |

**Rules:**
- `is_active = false` blocks login; treat as deleted for all API queries
- `password_hash` must never appear in any serialised API response
- `locked_until` checked before bcrypt comparison — return 423 if in future

## projects

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PK | Unique project identifier |
| name | VARCHAR(150) | NOT NULL | Project display name |
| description | TEXT | NULLABLE | Project description |
| owner_id | UUID | FK → users.id, NOT NULL | Project owner (manager or admin) |
| status | ENUM | DEFAULT 'active', NOT NULL | `active` \| `archived` \| `completed` |
| created_at | TIMESTAMPTZ | NOT NULL | — |
| updated_at | TIMESTAMPTZ | NOT NULL | — |

**Rules:**
- Only `manager` and `admin` roles can create projects
- Only `admin` can set `status = 'archived'`
- Archived projects are read-only; no new tasks can be created under them

## tasks

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PK | Unique task identifier |
| title | VARCHAR(200) | NOT NULL | Task title |
| description | TEXT | NULLABLE | Detailed task description |
| project_id | UUID | FK → projects.id, NOT NULL | Parent project |
| assignee_id | UUID | FK → users.id, NULLABLE | Assigned team member |
| reporter_id | UUID | FK → users.id, NOT NULL | Task creator |
| status | ENUM | DEFAULT 'todo', NOT NULL | `todo` \| `in_progress` \| `review` \| `done` |
| priority | ENUM | DEFAULT 'medium', NOT NULL | `low` \| `medium` \| `high` \| `critical` |
| due_date | DATE | NULLABLE | Task deadline |
| is_deleted | BOOLEAN | DEFAULT FALSE, NOT NULL | Soft-delete flag |
| created_at | TIMESTAMPTZ | NOT NULL | — |
| updated_at | TIMESTAMPTZ | NOT NULL | — |

**Rules:**
- `DELETE /tasks/:id` sets `is_deleted = true`; exclude from all queries by default
- Notification SVC fires async on `assignee_id` change or `status` change
- `due_date` triggers a `due_soon` notification 24 hours before via node-cron
- Tasks may not be created under a project with `status = 'archived'`

**Indexes** (in addition to PK):

| Index | Columns | Purpose |
|-------|---------|---------|
| Single | `project_id` | FK lookup |
| Single | `assignee_id` | FK lookup |
| Single | `reporter_id` | FK lookup |
| Single | `status` | Status filter |
| Single | `is_deleted` | Soft-delete filter |
| Single | `due_date` | Due-date range filter |
| Composite | `(project_id, status, is_deleted)` | Task list by project |
| Composite | `(assignee_id, status, is_deleted)` | Task list by assignee |
| Composite | `(due_date, status, is_deleted)` | Due-soon cron job query |

## comments

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PK | Unique comment identifier |
| task_id | UUID | FK → tasks.id, NOT NULL | Parent task |
| author_id | UUID | FK → users.id, NOT NULL | Comment author |
| body | TEXT | NOT NULL | Comment content (max 2000 chars enforced by Joi) |
| is_deleted | BOOLEAN | DEFAULT FALSE, NOT NULL | Soft-delete flag |
| created_at | TIMESTAMPTZ | NOT NULL | — |
| updated_at | TIMESTAMPTZ | NOT NULL | — |

**Rules:**
- `author_id` can soft-delete their own comments; admins can soft-delete any
- Posting a comment triggers a `task_commented` notification to the task assignee and reporter (if different from author)

---

## notifications

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PK | Unique notification identifier |
| recipient_id | UUID | FK → users.id, NOT NULL | Target user |
| type | ENUM | NOT NULL | `task_assigned` \| `task_updated` \| `task_commented` \| `task_due_soon` |
| reference_id | UUID | NOT NULL | ID of the entity that triggered the notification (task, comment) |
| reference_type | ENUM | NOT NULL | `task` \| `comment` |
| message | VARCHAR(500) | NOT NULL | Human-readable notification message |
| is_read | BOOLEAN | DEFAULT FALSE, NOT NULL | Read/unread state |
| created_at | TIMESTAMPTZ | NOT NULL | — |

**Indexes** (in addition to PK):

| Index | Columns | Purpose |
|-------|---------|---------|
| Composite | `(recipient_id, is_read, created_at DESC)` | Primary query: unread-first list per user |
| Single | `recipient_id` | FK lookup |

**Rules:**
- No `updated_at` — notifications are append-only
- `PATCH /notifications/:id` sets `is_read = true` (recipient only)
- `GET /notifications` returns unread-first, max 50 most recent
- Socket.io emits `notification:new` event to `recipient_id` room on insert

## Sequelize Migration Naming Convention

```
YYYYMMDDHHMMSS-create-<table-name>.js

e.g.
20260314000001-create-users.js
20260314000002-create-projects.js
20260314000003-create-tasks.js
20260314000004-create-comments.js
20260314000005-create-notifications.js
```

Run order matters due to FK constraints: `users` → `projects` → `tasks` → `comments` → `notifications`.
